# Deployment notes

This repo hosts **two** sites at the same GitHub Pages output:

| URL                                          | Source                                                |
|----------------------------------------------|-------------------------------------------------------|
| `schapiro.ai/`                               | Quarto-rendered from `*.qmd` → `docs/`                |
| `schapiro.ai/creative-ai-index/`             | Static Next.js export, committed to `docs/creative-ai-index/` |

Both live under `docs/` because GitHub Pages is configured to publish from `main` → `/docs`. The Creative AI Index export is **committed directly** to `docs/creative-ai-index/` — it is **not** regenerated by `quarto render`.

## Gotcha #1 — `quarto render` wipes `docs/creative-ai-index/`

Quarto's website project type cleans output paths it doesn't recognise. Since `_quarto.yml`'s `project.resources` does **not** list `creative-ai-index/**`, every `quarto render` deletes the entire `docs/creative-ai-index/` subtree from the working tree.

The deletions are unstaged. The committed copy is still in `HEAD`, so the live site is fine — until you stage those deletions.

**Why we don't add `creative-ai-index/**` to `resources:`** — it would force Quarto to copy ~500 files from the source-root `creative-ai-index/` into `docs/` on every render, taking 60+ seconds. The committed-directly approach is faster and keeps the Quarto build snappy.

## Workflow A — updating the Quarto personal site only

You changed a `.qmd`, e.g. you added a quote to `index.qmd`.

```bash
quarto render

# DO NOT run `git add -A` here — it would stage the deletion of every
# file under docs/creative-ai-index/.

git add _quarto.yml \
        docs/*.html docs/search.json docs/sitemap.xml docs/styles.css \
        index.qmd styles.css \
        # ... any other rendered Quarto files you touched

git commit -m "Render: <what you changed>"
git push origin main
```

Use **explicit paths**. Never `git add -A` or `git add .` after a bare `quarto render` — the staged deletions would erase the Creative AI Index from the next commit, taking the subsite offline until the next deploy.

The working tree after this still has `docs/creative-ai-index/` wiped on disk; that's fine, it will be restored next time you run the deploy script for the Index.

## Workflow B — updating the Creative AI Index

You changed something in the `creative-ai-index` repo (a new metric, a UI tweak, etc.).

```bash
# In the creative-ai-index repo, NOT this one:
cd ~/Desktop/creative-ai-index
npm run deploy:schapiro
```

`deploy:schapiro` does three things in order:
1. `next build` with the static-export config → produces `out/`.
2. `rm -rf` then `cp -R out/. ~/Desktop/samjschapiro2/creative-ai-index/` (source-root staging area).
3. `cd ~/Desktop/samjschapiro2 && quarto render` (regenerates Quarto pages).

Step 3 wipes `docs/creative-ai-index/`. Then **you need to copy the fresh export from the source-root into docs/** manually:

```bash
cd ~/Desktop/samjschapiro2
rm -rf docs/creative-ai-index
cp -R creative-ai-index docs/creative-ai-index

git add docs/creative-ai-index docs/*.html docs/search.json docs/sitemap.xml docs/styles.css
git commit -m "Update creative-ai-index static export (<what changed>)"
git push origin main
```

## Recovery — "I accidentally staged the creative-ai-index deletions"

Don't commit. Unstage the deletions, then continue with Workflow A:

```bash
git reset HEAD docs/creative-ai-index
git status   # confirm docs/creative-ai-index changes are no longer staged
```

If you've already committed the deletions but not pushed:

```bash
git reset --soft HEAD~1
git reset HEAD docs/creative-ai-index
# Re-stage what you actually wanted, then re-commit.
```

If you've already pushed, the live subsite is briefly offline — re-run Workflow B to restore it.

## Why this is the way it is

- We tried `project.resources: [creative-ai-index/**]` in `_quarto.yml` to make Quarto copy the export into `docs/` automatically. It works but slows every render to 60+ seconds because Quarto copies ~500 files on every invocation. Not worth it.
- Committing the export directly under `docs/creative-ai-index/` is the fastest path. The only cost is the discipline above.
